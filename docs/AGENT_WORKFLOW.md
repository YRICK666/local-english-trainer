# Agent Workflow

本文件是 `AGENTS.md` 的执行细则。后续 prompt 应引用本文件，只补充本轮目标、允许文件、已声明 dirty baseline、验收命令和特殊风险。

## 1. 每轮任务契约

一个任务只做一个能独立验证的小闭环。开始前记录：任务目标、允许修改范围、禁止范围、已声明的中间文件、是否允许 Git 写入、是否可触及临时数据根。不要把阶段末的完整回归、干净工作区或人工视觉确认自动变成实现开始条件。

推荐 prompt 结构：`在已声明 baseline 上完成 <单一目标>；允许 <文件>；验证 <命令/人工项>；遵守 AGENTS.md。`

## 2. 统一 preflight 与结束检查

开始时仅做一次轻量检查：

```powershell
git status --branch --short
git status --short
git diff --name-status
git diff --cached --name-status
git diff --check
```

- 已声明且在任务范围内的未提交修改必须登记并保留；它们不是未知污染。
- staged 内容必须有明确来源和当轮授权；未知 staged 内容必须停止。
- 结束时重跑上述检查，并确认修改仅在允许范围内。纯文档任务不必运行构建；代码任务按本轮验收运行最小编译/测试，完整回归只在小阶段或阶段末执行。

## 3. 停止与自动修复分流

必须停止并报告的情形仅有：

1. 来源不明的 staged 内容；
2. 任务范围外、无法归属的修改；
3. 动作可能访问、覆盖、迁移或破坏真实数据；
4. 需要未授权的高风险 Git、进程或系统操作；
5. 无法安全理解现有代码语义，继续可能造成破坏。

以下都是普通工程问题，必须先分析、换方法并重试：patch 或文本锚点失败、换行/编码差异、路径与命令问题、首次编译失败、单测失败、unused warning、非必要工具缺失或可解释的 SKIP。

处理顺序：完整读取目标文件和报错 → 按符号/结构定位 → 使用受控行级编辑或保留逻辑的受控整文件重写 → 立即读取、`git diff --check` → 重跑受影响验证。不得用 `#[allow(...)]`、跳过测试或扩大权限掩盖问题。

## 4. 写入与工作区

只写当轮明确允许的文件。优先使用受控 patch；工具不可用时可用标准 UTF-8 文件写入，但每次写后必须立即读取并检查 diff。不得做模糊递归替换、批量覆盖目录或擅自用备份覆盖当前文件。

遇到 stash 或备份哈希差异时，先比较字节、BOM/编码、换行标准化后的文本和 Git diff；仅在存在实际内容差异且已理解语义时提出修复方案。字节哈希不同本身不是代码损坏证据。

## 5. Git 与阶段状态

默认只读 Git。未经明确授权，不执行任何 Git 写入。获授权提交时：

- 先检查作者邮箱是否可推送；GitHub 私有邮箱保护（GH007）应在提交前避免，而不是在 push 后修改历史。
- 精确暂存列出的文件，禁止 `git add .`；每笔提交前运行 `git diff --cached --name-status` 与 `git diff --cached --check`。
- docs/workflow、功能与生成/配置变更按逻辑拆分提交；不要让多个完成阶段长期堆积在同一工作区。

报告必须区分：

- **功能完成**：实现与本轮验收通过；
- **可提交**：范围和验证已审查，但尚未 commit；
- **已提交**：存在本地 commit；
- **已推送**：远端分支已同步。

## 6. 数据、进程与环境

真实 SQLite、用户数据、原始使用仓库和项目测试临时根必须严格区分。测试只能使用本轮创建、路径已记录、可安全清理的临时根；不打开、读取、复制或迁移真实数据库。

不 broad-kill 或按名称结束 Python、WebView2、桌面应用或 sidecar。只关闭本轮保存的精确 child/PID。任何永久 PATH、代理、Java、ACL、系统安全和工具链变更都需要单独授权；Cargo/Tauri 临时环境仅存在于当前进程。

## 7. 验证语义

- **PASS**：命令或检查实际运行并满足明确断言。
- **SKIP**：因记录的前置条件未满足而未运行；必须说明原因，不能写成 PASS。
- **人工确认**：自动化无法可靠验证的窗口可见性、布局或交互；必须由用户真实会话确认，不能伪造。
- **smoke**：验证明确的启动、业务或退出链路；只在任务授权时启动进程，且不得替代单元测试。
- **warning**：仅在当轮要求消除、表明错误，或影响验收时阻塞；其余记录来源后继续。

构建、smoke 与人工视觉确认必须分别记录。实现中运行受影响的单测/编译；完整前后端、打包或 Windows 回归仅在阶段末或任务明确要求时运行。

## 8. 桌面阶段边界

React 不得获得 sidecar token、端口、任意 URL、HTTP method/path、child handle、数据库路径或 Windows 资源路径。`/health` 和 `/desktop/shutdown` 仅供 Rust lifecycle。P2.5 测试只使用临时数据根；P3 才能接入用户数据目录、bootstrap 和迁移；P4 才处理 NSIS/发布。

## 9. 推荐报告格式

简要列出：基线与范围、修改文件、实现结果、验证 PASS/SKIP/人工项、残留 warning、Git 状态、以及功能完成/可提交/已提交/已推送状态。若停止，说明触发的五类停止条件之一及需要的最小授权。
